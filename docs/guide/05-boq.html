BOQ
5 commands — parameter setup, schedule creation, quantity auto-fill, pricing, contract import

The BOQ panel implements a full Bill of Quantities workflow for Israeli construction projects. The pipeline runs in order: set up shared parameters, create reporting schedules, auto-fill quantities from geometry, calculate effective prices for composite items, and import contract sections from Excel. All 5 commands share a common set of 19 shared parameters in the "BIMIsrael_BOQ" group.

BOQ Setup Ribbon: BOQ → BOQ Setup
Solid

What It Does

Creates 19 shared parameters and binds them as instance parameters to all 18 payable categories. This is the foundation step — all other BOQ commands depend on these parameters existing.

Workflow

  1. Click BOQ Setup.
  2. Handler checks app.SharedParametersFilename for an existing shared param file.
  3. If not found, a SaveFileDialog lets the user create/select one.
  4. Opens the shared parameter file and gets or creates the "BIMIsrael_BOQ" group.
  5. Creates 19 parameters (14 core + 5 generic-model-only) with ReInsert.
  6. Binds each parameter as an instance binding to the appropriate categories.
  7. Reports success/error count for each parameter.

Solid

What It Does

Creates 10 Revit schedules covering BOQ reporting, payment status, quality assurance, and per-stage breakdowns. If a schedule name already exists, appends _v2, _v3, etc. (up to 99).

Workflow

  1. Click BOQ Schedules.
  2. Handler creates all 10 schedules in a single transaction.
  3. Each schedule gets fields, sort groups, filters, and grand totals configured automatically.

BOQ AutoFill Qty Ribbon: BOQ → BOQ AutoFill
Sketchy

What It Does

Auto-populates BI_QtyValue and BI_QtyBasis for all elements where BI_IsPayItem = Yes. Reads geometry from Revit, converts from imperial to metric, applies multipliers and overrides, and copies legacy Hebrew parameters to BI_ equivalents.

Workflow

  1. Click BOQ AutoFill.
  2. Collects all elements in 18 payable categories where BI_IsPayItem = 1.
  3. For each element:
    • Determines BI_QtyBasis from category mapping (if not already set).
    • Special case: type name containing "CONC" → VOLUME instead of AREA.
    • Reads quantity from built-in parameters; falls back to geometry extraction.
    • Converts to metric (m, m², m³) and rounds to 4 decimal places.
    • Applies BI_QtyMultiplier and BI_QtyOverride.
    • Copies legacy Hebrew params to BI_ equivalents.
    • Writes BI_HostElementId (for hosted elements like doors/windows/mullions).
  4. COMP items are silently skipped.
  5. Reports updated / skipped / errors.

Category → Default Basis Mapping

VOLUME Structural Foundations, Structural Columns, Structural Framing, Structural Framing Systems, Columns

Key Files

Commands/Handlers/BOQAutoFillQtyHandler.cs
Transaction
Single transaction: "BOQ Auto-Fill Quantities"
Unit Conversion
Feet → meters (0.3048), sq.ft → m² (0.0929), cu.ft → m³ (0.0283). Revit stores everything in imperial internally.
Suspicious Area Geometry Fallback: Divide by 6

When HOST_AREA_COMPUTED is unavailable, the fallback sums all geometry face areas then divides by 6: return ConvertToSquareMeters(totalArea / 6). This assumes a box with 6 equal faces (average = one face area), which is incorrect for non-cubic geometry like walls, sloped roofs, or L-shaped elements. The result will be wrong for most real-world cases.

Fragile Concrete Detection

Determines VOLUME basis by checking if the type name contains "CONC" (case-sensitive substring). This is locale-dependent and could match unintended type names (e.g., "CONCRETE_BLOCK" treated same as "CONCRETE_POUR"). Non-English projects would miss concrete elements entirely.

COMP Items Silently Skipped

Elements with BI_QtyBasis = "COMP" return false without logging. The user sees these in the "skipped" count with no explanation of why they were skipped. A brief note in the results dialog would help.

Legacy Hebrew Parameter Copy

Copies values from Hebrew-named parameters (סעיף תקציבי, תאור סעיף, קבלן משנה) to BI_ equivalents. This is a migration bridge from the old parameter naming convention. Should be removed once all projects are migrated.

BOQ Calc Price Ribbon: BOQ → BOQ Calc Price
Solid

What It Does

Calculates effective unit price for COMP (composite/lump-sum) PayLine items by dividing contract value by the total physical quantity of matching elements. This answers: "if we spread this contract line item across all physical elements, what's the cost per unit?"

Effective Unit Price Calculation
graph TD A["Collect physical pay items\n(non-COMP, QtyValue > 0)"] --> B["Group by BOQ code + basis\nSum quantities"] B --> C["Find all COMP PayLines\n(GenericModel, QtyBasis=COMP)"] C --> D["For each PayLine:\nContractValue = UnitPrice × QtyValue"] D --> E{"Physical items exist\nfor same code + basis?"} E -->|Yes| F["EffUnitPrice =\nContractValue ÷ SumPhysicalQty"] E -->|No| G["EffUnitPrice = 0"] F --> H["Write BI_ContractValue\n+ BI_EffUnitPrice"] G --> H style A fill:#1e3a5f,color:#fff style D fill:#7c3aed,color:#fff style F fill:#166534,color:#fff style G fill:#9a3412,color:#fff

Workflow

  1. Click BOQ Calc Price.
  2. Collects all physical (non-COMP) pay items, sums BI_QtyValue grouped by BOQ code + basis.
  3. Finds all COMP PayLines (GenericModel elements with BI_QtyBasis = "COMP").
  4. For each PayLine:
    • Reads BI_BOQ_Code, BI_AnalysisBasis, BI_UnitPrice, BI_QtyValue.
    • Calculates ContractValue = UnitPrice × QtyValue.
    • Looks up total physical qty for same BOQ code + analysis basis.
    • Calculates EffUnitPrice = ContractValue ÷ SumPhysicalQty.
  5. Writes BI_ContractValue and BI_EffUnitPrice to each PayLine.
  6. Reports per-PayLine results with breakdown.

Example

PayLine: BOQ code "03-02-001", COMP, AnalysisBasis=AREA, UnitPrice=5,000, QtyValue=2
→ ContractValue = 5,000 × 2 = 10,000 ILS

Physical items with same BOQ code: Item A (50 m²) + Item B (30 m²) = 80 m²

→ EffUnitPrice = 10,000 ÷ 80 = 125 ILS/m²

Key Files

Commands/Handlers/BOQCalcEffUnitPriceHandler.cs
Transaction
Single transaction: "Calculate Effective Unit Prices"
Validation
Rejects PayLines missing BOQ code, invalid analysis basis, or zero unit price. Each error is collected and reported individually.
Silent Zero Results

When no physical items match a PayLine's BOQ code + analysis basis, BI_EffUnitPrice is set to 0 and marked as success. The user won't know if the calculation was skipped due to missing data or if the effective price is genuinely zero. Should log or flag these cases.

No Type Parameter Fallback

Unlike AutoFill, this handler only checks instance-level parameters when collecting physical items. Elements with BOQ codes set on the type (rather than instance) will be missed, potentially underestimating the physical quantity denominator.

Contract Sections (Seifei Choze) Ribbon: BOQ → Contract Sections
Sketchy

What It Does

Imports contract line items from an Excel file into Revit key schedules. Creates or updates key schedules for 9 categories, binding 4 Hebrew-named shared parameters. Each Excel row becomes a key entry with name, item number, description, price, and subcontractor.

Workflow

  1. Click Contract Sections.
  2. Handler checks if required shared parameters are bound. If not, binds them in a first transaction.
  3. Opens a file dialog to select an .xlsx / .xls file.
  4. Reads the first worksheet (row 2+). Columns: KeyName, ItemNumber, Description, Price, Subcontractor.
  5. Stops on 5 consecutive empty rows or 10,000 rows max.
  6. For each of 9 target categories:
    • Gets or creates BIMTasks_Key_{Category} key schedule.
    • Ensures all 4 parameter fields exist in the schedule.
    • For each Excel row: finds existing key by name (case-insensitive) or inserts new row.
    • Sets all 5 fields on the key element.
  7. Reports results per category.

Key Files

Commands/Handlers/CreateSeifeiChozeHandler.cs
Transactions
Two transactions: "Bind BIMTasks Parameters" (if needed) and "Import Key Schedules".
Price Parsing
Handles double, int, long, float, decimal types. Strips currency symbols (₪, $, €). Tries invariant and current culture.
Weak Parameter Binding Validation

Only checks if the first and last required parameter are bound to the first and last target category. If middle parameters are missing on middle categories, it goes undetected. The binding phase is skipped and key schedule updates will fail silently for those categories.

Fragile Key Row Insertion

After inserting a new row into a key schedule, the handler finds the new element by searching for an unnamed/empty-name element in the collector. This assumes only one unnamed element exists at any time. If multiple rows are inserted or an existing element has no name, the wrong element may be updated.

Hebrew Parameter Names Hardcoded as Unicode

The 4 required parameter names are stored as Unicode escape sequences (\u05E1\u05E2\u05D9\u05E3...). This is a brittle locale-specific dependency. If the shared parameter file uses slightly different names (e.g., with/without spaces), the lookup fails silently.

No Excel Format Validation

Assumes the Excel file has exactly 5 columns in a specific order with no header validation. A differently structured file will silently misassign data.

Cross-Cutting Observations
  • The pipeline design is sound — Setup → Schedules → AutoFill → CalcPrice → Import is a logical progression. Each command builds on the previous one's output. Running them out of order will produce incomplete or empty results.
  • Israeli construction domain is baked in — Hebrew parameter names, ILS currency, 4 hardcoded work stages (FOUNDATIONS/FRAME/BUILDING/FINISHES), and BIMIsrael-specific naming conventions. Adapting this for international projects would require significant refactoring.
  • AutoFill is the weakest link — the divide-by-6 geometry fallback and fragile concrete detection are likely to produce wrong quantities in edge cases. The built-in parameter path (HOST_AREA_COMPUTED, etc.) is reliable for standard elements, but the geometry fallback needs rework.
  • Contract Sections has the most issues — weak binding validation, fragile row insertion, and hardcoded Hebrew names make it the most brittle command in the panel. It works for the specific use case it was built for, but any deviation from the expected format will cause silent failures.
  • Error handling is consistent — all 5 commands use per-element try/catch, collect errors, and report success/error counts. No command will abort the entire batch on a single element failure.
