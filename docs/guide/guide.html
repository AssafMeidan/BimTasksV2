Architecture Overview
Bird's eye view — how the plugin boots, isolates itself, and executes commands
Why Two Worlds?

Revit loads all add-ins into a shared default AssemblyLoadContext. Other plugins (Enscape, Dynamo, pyRevit) ship their own versions of common DLLs. If BimTasksV2 loaded Prism 9 and Unity directly, it would clash with whatever Prism/DryIoc/Newtonsoft version another add-in already loaded.

The solution: BimTasksV2 runs in an isolated ALC (BimTasksIsolated). Only a thin proxy layer lives in the default context — it holds no references to Prism, Unity, Serilog, or Vosk. The two worlds communicate through reflection and Revit's own API types (which are shared).

Default ALC Rule
BimTasksApp, RibbonBuilder, and all proxy commands must never reference Prism, Unity, Serilog, or Vosk types. Only Revit API types and System types are allowed.
Shared Types
Revit API DLLs (RevitAPI, RevitAPIUI), WPF framework assemblies, and all System.* assemblies resolve from the default context — both ALCs see the same types. This is what makes reflection calls possible.

Threading & Command Dispatcher
How WPF events safely reach the Revit API

Revit's API is single-threaded — it can only be called from the main thread during specific windows (inside IExternalCommand.Execute() or IExternalEventHandler.Execute()). WPF buttons, voice commands, and dockable panel interactions run on separate threads. The CommandDispatcherService bridges this gap.

When Is the Dispatcher Needed?
  • Needed: Floating toolbar button clicks, dockable panel actions, voice commands — anything triggered from WPF or a background thread.
  • Not needed: Ribbon button clicks — these already run inside IExternalCommand.Execute() on Revit's main thread.

Architecture Notes & Observations
Patterns to be aware of when working in this codebase
Reflection-Heavy Bridge

All communication between default and isolated ALC goes through reflection — no compile-time type safety. If a handler class is renamed or its Execute method signature changes, the error only surfaces at runtime when the user clicks the button. A shared interface assembly (loaded in both ALCs) could provide compile-time guarantees.

Service Resolution Inside Execute()

Handlers resolve dependencies via ContainerLocator.Container.Resolve<T>() inside their Execute() method (service locator pattern). This works but makes dependencies invisible — you can't tell what a handler needs without reading the method body. Constructor injection would be cleaner but requires changes to the proxy infrastructure.

Floating Toolbar Lifecycle

The floating toolbar window is created and cached in BimTasksBootstrapper as a field. Toggling shows/hides it. If the window is closed via the X button (by the user or Windows), the cached reference may point to a disposed window. The toolbar handler should check IsLoaded before toggling visibility.

What Works Well
  • ALC isolation is solid — no DLL conflicts reported with Enscape, Dynamo, or pyRevit.
  • The CommandDispatcherService pattern is clean and correctly handles the Revit threading model.
  • Three-phase initialization correctly respects Revit's registration windows.

Wall To Floors Ribbon: Walls → Wall To Floors
Sketchy

What It Does

Auto-adjusts selected walls' base offset and top to sit flush against the nearest floors above and below. Uses ReferenceIntersector to cast rays from wall endpoints to find floor surfaces.

Workflow

  1. Select one or more walls.
  2. Click Wall To Floors.
  3. Handler finds (or falls back to) the default {3D} view for ray casting.
  4. For each wall:
    • Casts rays downward from wall base → finds floor below → adjusts base offset.
    • Casts rays upward from wall top → finds floor above → adjusts top offset or unconnected height.
  5. Shows completion summary.

Key Files

Commands/Handlers/AdjustWallHeightToFloorsHandler.cs
No Per-Wall Error Handling

The main loop has no try-catch per wall. If one wall has invalid geometry or a null LocationCurve, the entire operation crashes. Each iteration should be wrapped individually.

Assumes Linear Walls Only

Ray casting uses start/end points of the wall curve. Curved walls will only have rays at their endpoints, missing the mid-arc. Not necessarily wrong, but worth knowing.

Split Wall Ribbon: Walls → Split Wall
Solid

What It Does

Splits a compound (multi-layer) wall into individual single-layer walls. Each layer becomes a separate wall, offset to its correct position. Hosted elements (doors, windows) are transferred to the designated target layer. After splitting, saves results to JSON for the Trim Corners follow-up command.

Workflow

  1. Select one or more compound walls (non-compound walls are filtered out).
  2. Click Split Wall.
  3. Dialog shows layer breakdown: function, material, thickness, and a type-assignment combo per layer.
  4. For each layer, choose an existing WallType or select Auto-Create (creates a new type matching the layer's thickness).
  5. Choose which layer should receive hosted elements (Auto = thickest structural layer, or manual pick).
  6. Choose scope: selected walls only, or all walls of the same type in the project.
  7. Click OK — engine runs a 4-transaction pipeline per wall.
  8. Results saved to split_results.json. Summary dialog instructs you to run Trim Corners next.

CompoundLayerAnalyzer — How Offsets Work

Each layer's center offset is computed from the wall's centerline (ignoring WALL_KEY_REF_PARAM). Formula: centerOffset = (totalWidth/2) - (runningDistance + thickness/2). Positive offset = toward exterior = wall.Orientation direction. Zero-thickness membrane layers are skipped.

Revit Gotcha
TransactionGroup.Assimilate() invalidates in-memory Element references. The engine stores ElementId values and re-fetches after assimilation.
File Dependency
Results are saved to split_results.json in the plugin DLL directory. If the directory is not writable (e.g., locked by another process), the save fails silently and Trim Corners won't work.
Trim Corners Dockable Panel → Trim Corners button
Solid

What It Does

The second step of the Split Wall pipeline. After walls are split into layers, the replacement walls don't connect at corners. This command reads the JSON output from Split Wall and cross-joins replacement walls at their intersection points.

Workflow

  1. Run Split Wall first (produces split_results.json).
  2. Click Trim Corners in the dockable panel.
  3. Handler reads JSON, re-fetches walls from document (validates they still exist).
  4. Two-pass corner trimming:
    • Pass 1: Join replacement walls between split groups by matching layer index (outer↔outer, inner↔inner).
    • Pass 2: Trim each replacement against nearby unsplit neighbor walls.
  5. JSON file is deleted after success.

Key Files

Commands/Handlers/TrimWallCornersHandler.cs Helpers/WallSplitter/WallJoinReplicator.cs Views/TrimCornersView.xaml
Corner Detection Algorithm

Uses 2D line-line intersection on wall centerlines. An intersection is considered a "corner" if the intersection point is within 500mm of an endpoint on both walls. Parallel walls or distant intersections are ignored. End-join states are toggled (allow→disallow→allow) to force geometry refresh.

Dependency
Completely depends on split_results.json from the Split Wall step. If the file is missing, shows "Run Split Wall first."
Thread Safety
Runs via CommandDispatcherService (queued from the dockable panel button), not directly from a ribbon command.
Detect Overlaps Ribbon: Walls → Detect Overlaps
Solid

What It Does

Finds walls of the same type that overlap each other. Groups them, calculates overlap percentage, and lets you highlight, unjoin, or delete the duplicates. 100% overlaps (exact duplicates) sort to the top.

Workflow

  1. Optionally select walls to scan (if none selected, scans all walls in active view).
  2. Click Detect Overlaps.
  3. Detector groups walls by type, then checks pairwise:
    • Are they parallel? (angular tolerance ~1°)
    • Is perpendicular distance < 50% of wall width?
    • Same base level?
    • Overlap length ≥ 50mm?
  4. Overlap % computed via 3D solid boolean intersection (falls back to 2D estimate if geometry fails).
  5. Results dialog shows groups with: overlap%, type, wall count, IDs, and suggested "Keep" wall.
  6. Choose action:
    • Highlight: Selects and temporarily isolates overlapping walls.
    • Unjoin: Unjoins all pairs in each group.
    • Delete Duplicates: Keeps the wall with the most hosted elements, deletes the rest.

Sketchy

What It Does

Adds cladding (chipuy) walls parallel to selected walls. Three variants in a split button: Both Sides, External Only, and Internal Only. Each creates offset walls using predefined cladding wall types, then joins geometry and connects end-join corners.

CladdingWallCreator resolves cladding types by name ("w1" for external, "w2" for internal). If these types don't exist in the project, it throws InvalidOperationException with no recovery path. Should either prompt for type selection or create the types automatically.

Beams To Ground Ribbon: Structure → Beams To Ground
Sketchy

What It Does

Recreates selected beams at Level 0 while preserving their top elevation. Useful when beams are modeled at arbitrary levels but need to reference Level 0 with a Z-offset for structural analysis.

Workflow

  1. Select one or more structural framing elements (beams).
  2. Click Beams To Ground.
  3. Handler finds Level 0 (elevation == 0) in the document.
  4. For each beam:
    • Reads the beam's top elevation (STRUCTURAL_ELEVATION_AT_TOP).
    • Creates a new beam at Level 0 with the same curve and family type.
    • Sets Z-justification to Top, reference level to Level 0.
    • Sets Z-offset to the original top elevation (preserves visual position).
  5. Deletes all original beams in batch.
  6. Shows count summary.

Key Files

Commands/Handlers/SetBeamsToGroundZeroHandler.cs
Originals Deleted Before Verifying Replacements

The handler deletes original beams after creating replacements, but does not verify that each new beam was actually created successfully. If doc.Create.NewFamilyInstance() returns null (rare but possible), the original beam is deleted with no replacement. Should track which originals have confirmed replacements before batch-deleting.

No Try-Catch in Main Loop

Relies on null checks but has no exception handling around individual beam processing. One malformed beam geometry could abort the entire operation mid-way, leaving some beams duplicated and others not yet processed.

Join Walls & Floors Ribbon: Structure → Join Walls & Floors
Solid

What It Does

Joins geometry between all concrete walls and the selected floors, roofs, ceilings, or structural foundations. Uses ElementIntersectsElementFilter to find spatially overlapping walls, then joins each pair. Handles join-order sensitivity by retrying in reverse order on failure.

Workflow

  1. Select one or more floors, roofs, ceilings, or foundations.
  2. Click Join Walls & Floors.
  3. For each selected host element:
    • Finds all walls that intersect it geometrically.
    • Skips: already-joined pairs, thin walls (<30mm), curtain walls, stacked walls.
    • Attempts JoinGeometry(host, wall).
    • If ArgumentException, retries with switched order: JoinGeometry(wall, host).
  4. Shows detailed results: walls joined (normal vs. switched), skipped, already joined, failed (with IDs).

Key Files

Commands/Handlers/JoinAllConcreteWallsAndFloorsGeometryHandler.cs
Design Strength
Excellent per-element error handling. Every operation is wrapped in try-catch. Failures are logged and reported but never stop the overall process. The switched-order retry handles Revit's unpredictable join-order sensitivity.
Warning Suppression
Uses SuppressWarningsPreprocessor to silently handle Revit's geometry join warnings that would otherwise halt the transaction.

Cross-Cutting Issues
Architectural debt, systemic patterns, and improvement opportunities

This chapter collects issues that span multiple commands or affect the add-in's architecture as a whole. Per-command issues are documented inline in their respective chapters (2–6). These are the patterns that emerged across the full codebase review.

ALC Isolation Fragility
Architectural

The Problem

The entire add-in runs inside an AssemblyLoadContext("BimTasksIsolated") to keep Prism 9, Unity, Vosk, and other dependencies away from Revit's default ALC. The thin proxy layer in BimTasksApp.cs uses reflection to cross the boundary.

This works, but creates several fragile seams:

  • Static fields are per-ALC. A static property set in the default ALC (e.g., BimTasksApp.DockablePaneId) reads as null from the isolated ALC. Every cross-ALC reference must go through reflection or shared interfaces.
  • WPF bindings may not update source when the DataContext lives in the isolated ALC. HierarchicalDataTemplate bindings are particularly affected — explicit code-behind event handlers are needed as workarounds.
  • Type identity breaks across ALCs. An IExternalCommand loaded in the isolated ALC is not the same type as IExternalCommand in the default ALC. This is why proxy commands exist at all.
  • Debugging is harder. Stack traces show reflection invoke boundaries. Breakpoints in isolated-ALC code require the debugger to attach to the correct context.

Affected Components

BimTasksApp.cs (default ALC proxy) Infrastructure/BimTasksBootstrapper.cs (isolated ALC) Commands/Proxies/*.cs (all proxy commands)
Recommendation

The ALC pattern is necessary but underdocumented. Consider adding a ARCHITECTURE.md at the repo root explaining the ALC boundary, what can/cannot cross it, and the proxy pattern rules. New commands will inevitably hit these issues without guidance.

Silent Failure Patterns
Systemic

The Problem

Several commands swallow exceptions or return default values without notifying the user. The per-element try/catch pattern is good for resilience (one bad element shouldn't abort the batch), but the reporting side often loses the detail:

  • BOQ Schedules: Empty try/catch in FindField(), AddSortGroup(), AddFilterHasValue(). Schedules appear correct but may have no sorting or filtering.
  • BOQ AutoFill: COMP items silently skipped (no log, no explanation in results). Geometry fallback returns 0 without warning.
  • BOQ Calc Price: EffUnitPrice = 0 marked as success when no matching physical items found. User can't distinguish "genuinely zero" from "data missing."
  • Contract Sections: Middle parameters may be unbound (only first/last checked). Key row insertion may update wrong element.
  • Export JSON: Category comparison silently skips if schedule name doesn't match mapping.
Recommendation

Add a "warnings" section to results dialogs. Currently most commands report "X succeeded, Y failed" but no "Z completed with caveats." A three-tier result (success / warning / error) would surface these silent issues without blocking the workflow.

Transaction Patterns & Consistency
Observation

What's Good

Transaction discipline is consistent across the codebase:

  • All handlers use [Transaction(TransactionMode.Manual)].
  • Complex workflows use TransactionGroup with Assimilate() (Split Wall's 4-transaction pipeline, Filter to Legend's create + populate).
  • Batch operations commit if ≥1 success, rollback if all fail.
  • Read-only commands (Filter Tree, Calc Area, Export JSON) correctly avoid transactions.

Areas of Concern

  • TransactionGroup.Assimilate() invalidates Element refs — code must re-fetch by ElementId after assimilation. Most code handles this, but it's a recurring source of bugs when adding new features to multi-transaction commands.
  • Copy From Link uses a different pattern (batch → chunked fallback) that creates per-batch transactions in the fallback path. This means partial results are possible (some chunks committed, some failed).
Error Handling & Reporting
Observation

What's Good

  • Per-element try/catch throughout — one bad element never aborts the batch.
  • Serilog logging at every major step (Information for flow, Warning for recoverable, Error for fatal).
  • TaskDialog for user-facing results with success/error counts.
  • OperationCanceledException caught gracefully (user pressed Esc).

What's Inconsistent

  • Some handlers use TaskDialog.Show() directly, others use helper methods. No shared result dialog component.
  • Error message detail varies wildly — Export JSON shows file paths and validation breakdown; Piles From DWG shows just a count.
  • Serilog log levels are inconsistent — some handlers log expected conditions (e.g., "no elements selected") as Warning, others as Information.
Recommendation

Create a shared CommandResult class with success/warning/error counts and a ShowResultDialog() method. This would standardize reporting across all handlers and make the three-tier result pattern easy to adopt.

Element Reference Invalidation
Systemic

The Problem

Revit Element objects become invalid after transactions that modify or delete them. TransactionGroup.Assimilate() also invalidates in-memory references. Accessing properties on an invalid element throws InvalidOperationException.

Current Mitigations

  • Element.IsValidObject checks before property access (used in Wall Splitter, Detect Overlaps).
  • Store ElementId instead of Element and re-fetch after transactions (used in Filter Tree).
  • Re-collecting elements after Assimilate() (used in Split Wall pipeline).

Where It's Not Handled

  • Detect Overlapping Walls: stores wall list (not IDs) — safe only because all operations are within a single transaction, but if the transaction model changes, walls could go stale.
  • Contract Sections: key element references retrieved after row insertion may be invalid if Revit's internal schedule restructuring runs between operations.
Recommendation

Adopt a consistent "store ElementId, re-fetch on use" pattern across all handlers. The current mixed approach (some store Elements, some store IDs) is a maintenance risk.

Service Resolution Timing
Architectural

The Problem

Services are registered in BimTasksBootstrapper.OnFirstIdle() (phase 3 of startup). ViewModels that resolve services in their constructors will fail if constructed before OnFirstIdle() fires. The dockable panel ViewModel was explicitly designed to avoid this, but new ViewModels could easily fall into the trap.

Current Mitigation

BimTasksDockablePanelViewModel uses lazy resolution — services are resolved on first use, not in the constructor. This works but is not enforced by the framework.

Recommendation

Document the "no constructor resolution" rule prominently. Consider a base ViewModel class that provides lazy service accessors, making the pattern easy to follow for new code.

Floating Toolbar — Known Broken
Broken

The Problem

The floating toolbar is the only fully broken feature in the add-in. It has not been debugged yet. Likely causes:

  • Window lifecycle: Singleton pattern (cancel close → hide) may conflict with WPF's window state in the isolated ALC.
  • Event subscription timing: ToggleFloatingToolbarEvent may fire before the subscriber is wired up.
  • ALC boundary: WPF modeless windows in isolated ALCs have known issues with owner assignment and dispatcher affinity.
  • Command dispatch: Toolbar buttons need to invoke commands through ICommandDispatcherService, which requires Revit API thread access. If the dispatch path is broken, buttons appear but do nothing.
Debugging Plan

1. Verify ToggleFloatingToolbarEvent reaches the subscriber (add Serilog trace).
2. Check if the window is created and shown (may be off-screen or behind Revit).
3. Test button click → ICommandDispatcherService.Enqueue() path.
4. If window shows but buttons fail, the ALC command dispatch path is broken.
5. If window doesn't show, check WPF owner/dispatcher affinity in isolated ALC.
