Architecture Overview
Bird's eye view — how the plugin boots, isolates itself, and executes commands
Why Two Worlds?

Revit loads all add-ins into a shared default AssemblyLoadContext. Other plugins (Enscape, Dynamo, pyRevit) ship their own versions of common DLLs. If BimTasksV2 loaded Prism 9 and Unity directly, it would clash with whatever Prism/DryIoc/Newtonsoft version another add-in already loaded.

The solution: BimTasksV2 runs in an isolated ALC (BimTasksIsolated). Only a thin proxy layer lives in the default context — it holds no references to Prism, Unity, Serilog, or Vosk. The two worlds communicate through reflection and Revit's own API types (which are shared).

Default ALC Rule
BimTasksApp, RibbonBuilder, and all proxy commands must never reference Prism, Unity, Serilog, or Vosk types. Only Revit API types and System types are allowed.
Shared Types
Revit API DLLs (RevitAPI, RevitAPIUI), WPF framework assemblies, and all System.* assemblies resolve from the default context — both ALCs see the same types. This is what makes reflection calls possible.

Why Three Phases?
  • Phase 1 (Initialize): No UIApplication available yet — only UIControlledApplication. Can register services and build UI, but cannot create ExternalEvent or access active documents.
  • Phase 2 (Dockable Pane): Must happen during OnStartup — Revit does not allow pane registration after the startup window closes.
  • Phase 3 (OnFirstIdle): First time we have a real UIApplication. ExternalEvent.Create() must be called on the Revit main thread, so this is where CommandDispatcherService gets created.

Every ribbon button maps to a proxy command in the default ALC. The proxy is a one-liner — it just declares the handler's class name. All actual logic lives in the handler inside the isolated ALC.

// Default ALC — proxy command (one per ribbon button)
[Transaction(TransactionMode.Manual)]
public class SplitWallCommand : ProxyCommand
{ protected override string HandlerTypeName => "SplitWallHandler"; }

// Isolated ALC — the real handler
public class SplitWallHandler : ICommandHandler
{ public void Execute(UIApplication app) { /* all the work */ } }
Gotcha
Handlers are created via Activator.CreateInstance() — they must have a parameterless constructor. Dependencies are resolved inside Execute() via ContainerLocator.Container.Resolve<T>().

1.3
Threading & Command Dispatcher
How WPF events safely reach the Revit API

Revit's API is single-threaded — it can only be called from the main thread during specific windows (inside IExternalCommand.Execute() or IExternalEventHandler.Execute()). WPF buttons, voice commands, and dockable panel interactions run on separate threads. The CommandDispatcherService bridges this gap.

Revit->>Queue: Drain all queued actions Queue-->>Revit: action(UIApplication) Revit->>Revit: Execute on main thread
When Is the Dispatcher Needed?
  • Needed: Floating toolbar button clicks, dockable panel actions, voice commands — anything triggered from WPF or a background thread.
  • Not needed: Ribbon button clicks — these already run inside IExternalCommand.Execute() on Revit's main thread.

Event System
Prism EventAggregator for decoupled UI coordination

Handlers and ViewModels communicate through Prism's EventAggregator — publishers don't need a reference to subscribers. Four events are defined:

Event Payload Published By Consumed By
ToggleFloatingToolbarEvent object ToggleToolbarHandler BimTasksBootstrapper (creates/toggles FloatingToolbarWindow)
SwitchDockablePanelEvent string (view name) ShowFilterTreeWindowHandler, CheckSelectedWallsHandler BimTasksDockablePanelViewModel (navigates region)

Prism 9 Threading Gotcha
In Prism 9, ThreadOption.UIThread is async (unlike Prism 8). For synchronous delivery, use ThreadOption.PublisherThread. The floating toolbar subscription uses this to avoid race conditions.

Key Services
Singleton services registered in the Unity container
RevitContextService
Holds UIApplication (set in OnFirstIdle). UIDocument is a computed property that always returns the active document — never cached, because the user can switch documents.
CommandDispatcherService
Thread-safe action queue using ExternalEvent. Bridges WPF/voice threads to the Revit main thread. See section 1.3.
ScheduleExcelExportService
Professional Excel export using EPPlus. Handles multi-schedule export (single file or separate files), column formatting, merged headers, and auto-fit. Used by the Export Schedule command.
ScheduleExcelRoundtripService
Exports a schedule to a temp Excel file, opens it for editing, then reimports changes on user confirmation. Used by the Edit In Excel command.
DataExtractorService
Extracts all model elements into BimElementDto objects and serializes to JSON. Includes validation comparison against Revit schedule totals. Used by the Web App Data Extractor command.
VoskVoiceCommandService
Offline voice recognition using Vosk + NAudio. Grammar-constrained to recognize command names. Optional — creation is wrapped in try/catch so the plugin works even if Vosk model files are missing.
ElementCalculationService
Calculates area and volume for selected Revit elements. Results are displayed in the dockable panel's ElementCalculationView.
UniformatDataService
Provides Uniformat classification code lookups. Displayed in a modeless window for reference.

1.7
Architecture Notes & Observations
Patterns to be aware of when working in this codebase
Reflection-Heavy Bridge

All communication between default and isolated ALC goes through reflection — no compile-time type safety. If a handler class is renamed or its Execute method signature changes, the error only surfaces at runtime when the user clicks the button. A shared interface assembly (loaded in both ALCs) could provide compile-time guarantees.

Service Resolution Inside Execute()

Handlers resolve dependencies via ContainerLocator.Container.Resolve<T>() inside their Execute() method (service locator pattern). This works but makes dependencies invisible — you can't tell what a handler needs without reading the method body. Constructor injection would be cleaner but requires changes to the proxy infrastructure.

Floating Toolbar Lifecycle

The floating toolbar window is created and cached in BimTasksBootstrapper as a field. Toggling shows/hides it. If the window is closed via the X button (by the user or Windows), the cached reference may point to a disposed window. The toolbar handler should check IsLoaded before toggling visibility.

What Works Well
  • ALC isolation is solid — no DLL conflicts reported with Enscape, Dynamo, or pyRevit.
  • The CommandDispatcherService pattern is clean and correctly handles the Revit threading model.
  • Three-phase initialization correctly respects Revit's registration windows.
